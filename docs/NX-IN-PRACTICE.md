# Nx in this workspace, in practice

This repo is a single Nx monorepo (`@workspace/source`) generated by `create-nx-workspace`,
using **Bun** as the package manager and **mise** as the task runner shim. Right now it holds
one real app (`api`, a Fastify server) and its e2e test project. Nx itself is not a framework —
here it's doing three concrete jobs:

## 1. Project + target inference

There's no hand-written `project.json`. Targets (`build`, `serve`, `test`, `docker:build`, etc.)
come from two places:

- **Inferred by plugins** — `nx.json` registers `@nx/vite` and `@nx/docker` plugins, which scan
  for config files (`vite.config.ts`, `Dockerfile`) and auto-create targets from them.
- **Declared in `package.json` under the `"nx"` key** — see `apps/api/package.json`, which
  defines `build` (via `@nx/esbuild:esbuild`), `serve` (via `@nx/js:node`, depends on `build`),
  `test` (via `@nx/jest:jest`), and `prune`/`copy-workspace-modules` (packaging steps for the
  Docker image).

Run `npx nx show project api` to see the fully resolved target graph for a project — that's the
actual source of truth, not the config files individually.

## 2. Task orchestration + caching

Nx's value here is dependency-aware task running: `serve` depends on `build`, `docker:build`
depends on `build` + `prune`, etc. Nx resolves that graph and (per `namedInputs` in `nx.json`)
caches task outputs based on which files actually changed — a `production` input set excludes
spec files and test-setup so unrelated test edits don't invalidate build cache.

In practice nobody runs `nx` directly day-to-day — `mise.toml` wraps the common ones:

```
mise run serve   -> bunx nx serve @workspace/api
mise run build   -> bunx nx build @workspace/api
mise run test    -> bunx nx test @workspace/api
mise run e2e     -> bunx nx e2e @workspace/api-e2e
```

and `package.json` scripts (`start`/`build`/`test`) just call those mise tasks. So the practical
entrypoint is `mise run <task>` or `npm start`, with Nx doing the actual dependency-ordered
execution underneath.

## 3. Nx Cloud

`nx.json` has an `nxCloudId`, and git history shows a dedicated `feat(nx-cloud): setup nx cloud
workspace` commit. This wires remote caching (and potentially distributed task execution) so CI
runs and other machines can reuse cached task outputs instead of rebuilding/retesting from
scratch.

## What's NOT really in use yet

- Only one app + one e2e project exist (`apps/api`, `apps/api-e2e`) — this is still close to
  scaffold state, not a large multi-app monorepo where Nx's affected-graph/dependency-boundary
  features would really pay off.
- `AGENTS.md`/`CLAUDE.md` reference an `nx-workspace` skill and an `nx-generate` skill for
  agent-driven exploration/scaffolding, but `.claude/skills/` doesn't exist in the checkout
  (gitignored — likely provisioned via the Nx MCP server / Nx Console rather than committed).
- No custom Nx plugin or generator has been authored — everything comes from stock
  `@nx/vite`, `@nx/docker`, `@nx/esbuild`, `@nx/jest`, `@nx/js` executors.

## Bottom line

In this workspace, Nx is a **task graph + cache layer** sitting between `mise`/Bun and the
actual tools (esbuild, Jest, Docker, Vite's inferred targets). It doesn't replace those tools —
it just gives them dependency ordering, incremental caching, and (via Nx Cloud) shared remote
cache across machines/CI. The day-to-day developer experience is still "run a mise task"; Nx is
the thing making repeated `build`/`test`/`docker:build` runs fast and consistent underneath that.
